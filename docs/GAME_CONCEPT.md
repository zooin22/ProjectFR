# ProjectFR - Game Concept

## High Concept

플레이어는 **해커 오퍼레이터**다.
의뢰인에게서 계약을 받고, 폴더 구조로 시각화된 시스템에 침투해 **자료 회수, 삭제, 정찰, 변조** 같은 임무를 수행한다.

전투는 단순한 몬스터 처치가 아니라, 파일 시스템 내부에서 벌어지는 **작전 수행 과정**이다.
폴더는 침투 구역이고, 파일과 특수 파일은 보안 자산·핵심 데이터·방해 요소다.

---

## Core Fantasy

- 위험한 시스템에 잠입한다
- 특정 타깃 파일이나 폴더를 처리한다
- 들키기 전에 빠져나온다
- 보상을 받고 더 큰 의뢰를 수주한다
- 실패하면 자금, 평판, 추적도에 타격을 입는다

핵심 감정은:
**"전투를 이기는 것"보다 "작전을 성공시키는 것"** 이다.

---

## Player Role

플레이어는 현장형 해커다.
직접 총을 쏘는 전사가 아니라,
- 파일을 열고
- 삭제하고
- 복사하고
- 붙여넣고
- 압축하고
- 격리하면서
시스템 내부를 교란한다.

액션은 곧 해킹 툴셋이다.

---

## Core Loop

1. **의뢰 선택**
   - 의뢰인, 목표, 제한 턴, 보상, 실패 리스크 확인
2. **침투 시작**
   - 폴더 던전에 진입
3. **작전 수행**
   - 목표 파일/폴더 탐색
   - 전투와 상태이상, 경로 정리
   - 목표 달성
4. **탈출/정리**
   - 남은 보안 개체를 정리하거나 안전 경로 확보
5. **정산**
   - 성공 시 크레딧/평판 획득
   - 실패 시 크레딧 손실/추적도 증가
6. **다음 계약**
   - 더 위험하거나 더 복잡한 의뢰 해금

## Dungeon Interpretation

### Folder = 구역 / 방 / 보안 계층
- 루트 폴더: 침투 시작점
- 하위 폴더: 더 깊은 구역
- 특정 폴더: 보안 성격이 다른 테마 존

### File = 일반 대상 / 보안 개체
- 삭제 대상
- 방해 요소
- 미끼 데이터

### Special File = 핵심 타깃 / 보스급 보안 자산
- 회수 대상 자료
- 감추어진 payload
- 강한 방어/위험 요소

---

## Mission Structure

각 미션은 최소한 아래 요소를 가진다.

- 미션 이름
- 의뢰인
- 브리핑
- 목표 타입
- 목표 경로
- 제한 턴
- 성공 보상
- 실패 페널티

### Objective Types
- **Extract**: 특정 파일을 회수
- **Delete**: 특정 파일/폴더를 삭제
- **Scan**: 특정 노드를 정찰/분석
- 추후 확장:
  - Sabotage
  - Replace
  - Escort Data
  - Timed Escape

---

## Reward / Penalty Economy

### Rewards
- **Credits**: 장비, 업그레이드, 소모품 구매
- **Reputation**: 상위 계약 해금, 의뢰인 신뢰도 상승
- 추후 확장:
  - 특수 액션 해금
  - 시작 버프
  - 클립보드 기반 특수 툴

### Failure Penalties
- **Credits Loss**
- **Heat Increase** (추적도)
- 평판 하락
- 추후 확장:
  - 특정 의뢰인 차단
  - 보안 강화 미션 증가
  - 다음 작전 시작 디버프

---

## Heat / Trace Concept

추적도(Heat)는 이 세계관의 압박 장치다.

- 작전 실패
- 제한 턴 초과
- 소란스러운 침투
- 위험 대상 처리 실패

이런 행동이 누적되면 Heat가 오른다.

Heat가 높아지면 추후:
- 적 AP 증가
- 적 HP/공격력 증가
- 특수 보안 적 등장
- 제한 턴 감소
- 보상 대비 위험도 상승

현재 프로토타입에는 이미 아래 효과가 들어간다.
- **Heat 0-2:** 기본 상태
- **Heat 3-5:** 제한 턴 -1, 적 HP +1, 적 ATK +1
- **Heat 6+:** 제한 턴 -2, 적 HP +3, 적 ATK +2, 적 AP +1

즉 실패는 단순 재시작이 아니라 **캠페인 리스크**가 된다.

---

## Tone & Narrative Direction

톤은 가볍게 시작하되, 점점 불편하고 음성적인 세계로 깊어질 수 있다.

예시 방향:
- 초반: 기업 문서 절도, 로그 삭제, 캐시 정리
- 중반: 내부 고발 자료 확보, 감염 파일 제거, 경쟁사 sabotage
- 후반: 정치/군사/AI 통제 시스템 침투, 진실 은폐 또는 폭로

스토리는 미션 단위로 쌓는 구조가 잘 맞는다.

### Good Story Hooks
- 의뢰인이 항상 진실을 말하지 않는다
- 같은 파일도 누구 입장에선 증거, 누구 입장에선 생명줄이다
- 단순 삭제 의뢰가 사실은 진실 은폐일 수 있다
- 회수한 자료를 넘길지, 숨길지, 공개할지 선택할 수 있다

## Factions / Clients Ideas

- **Northline Legal**: 증거 삭제, 기록 정리, 컴플라이언스 은폐
- **Morrow Proxy**: 데이터 탈취, 연구 자료 회수, 기업 스파이 라인
- **Helix Ops**: 정찰, 침입 구조 분석, 보안 평가 위장 의뢰
- **Ember Circuit**: 위험하지만 고수익인 암시장 브로커
- **Glass Key Collective**: 진실 폭로형 회수 의뢰

이 구조는 나중에 스토리라인, 분기, 평판 시스템으로 확장하기 좋다.
자세한 초안은 `docs/MISSION_FACTIONS.md`에 정리한다.

---

## Design Pillars

1. **작전 중심**
   - 단순 전투 승리보다 목표 달성이 중요
2. **파일 시스템 판타지 유지**
   - 모든 액션과 적, 구역이 파일/폴더 문법 안에서 설명되어야 함
3. **짧고 강한 전술 판단**
   - AP, 상태이상, 타깃 우선순위, 턴 제한 압박
4. **미션 결과의 지속성**
   - 성공과 실패가 다음 계약에 영향을 줘야 함
5. **스토리 확장 가능성**
   - 미션 브리핑만 읽어도 세계관 떡밥이 쌓여야 함

---

## Recommended Near-Term Implementation Order

1. 미션 데이터 구조 고정
2. 미션 성공/실패 판정 안정화
3. 크레딧/평판/추적도 루프 정착
4. 미션 보드 확장
5. 폴더별 특수 규칙 추가
6. 스토리 훅/의뢰인 라인 추가
7. 업그레이드/상점/영구 성장 연결

## Current Prototype Interpretation

현재 프로토타입은 다음과 같이 해석할 수 있다.

- 메인 메뉴 = 계약 선택 화면
- 폴더 던전 = 단일 시스템 침투 맵
- 액션 전투 = 해킹 툴 운용
- 전투 종료 = 작전 정산
- 미션 보드 = 캠페인의 최소 시작점

즉 다음 단계 목표는
**"전투 프로토타입"을 "작전 기반 해커 미션 게임"으로 변환하는 것** 이다.
