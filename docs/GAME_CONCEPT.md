# ProjectFR - Explorer-Based Game Concept

## 1. High Concept

**ProjectFR**는 윈도우 파일 탐색기 조작을 게임 규칙으로 재해석한 **파일 작업 기반 전술 로그라이크**다.

플레이어는 해커 오퍼레이터가 되어 의뢰를 받고, 실제 파일 탐색기처럼 보이는 시스템 내부에 침투한다. 목표는 적을 처치하는 것이 아니라, 파일을 회수하고, 삭제하고, 숨기고, 변조하고, 로그를 위조하며, 보안 모듈의 추적을 피해 작전을 완수하는 것이다.

핵심 차별점은 다음 한 문장으로 정리된다.

> 전투 시스템을 따로 만드는 것이 아니라, 윈도우 탐색기 조작 자체를 전투 시스템으로 만든다.

---

## 2. Core Fantasy

플레이어가 느껴야 하는 핵심 판타지는 다음과 같다.

- 위험한 시스템에 몰래 접속한다.
- 폴더와 파일을 실제 탐색기처럼 조작한다.
- 목표 파일을 찾아 회수하거나 삭제한다.
- 보안 모듈의 감시를 피하거나 교란한다.
- 흔적을 지우거나, 일부러 가짜 흔적을 남긴다.
- 매 런마다 다른 빌드와 다른 방식으로 작전을 수행한다.
- 실패해도 다음 런에서 더 많은 도구와 전략으로 다시 시도한다.

핵심 감정은 다음과 같다.

> 적을 이기는 것이 아니라, 시스템을 속여서 작전을 성공시키는 것.

---

## 3. Genre Definition

ProjectFR의 장르 방향은 다음과 같다.

- 파일 탐색기 기반 전술 로그라이크
- 침투/잠입 전략 게임
- 파일 작업 진행률 중심의 작전 게임
- 빌드 중심 해킹 로그라이크
- 캐릭터 표현이 있는 탐색기형 전투 UI

ProjectFR의 핵심 흐름은 다음에 가깝다.  
선택 → 명령 → 파일 작업 진행 → 보안 반응 → 흔적 관리 → 탈출

플레이어는 공격 버튼을 누르는 것이 아니라, 실제 탐색기 사용자처럼 파일과 폴더에 명령을 내린다.

---

## 4. Design Pillars

### 4.1 탐색기 조작이 곧 게임 규칙

게임의 모든 핵심 행동은 실제 파일 탐색기에서 볼 수 있는 조작에서 출발해야 한다.

예시:

- 열기
- 복사
- 붙여넣기
- 이동
- 삭제
- 이름 바꾸기
- 압축
- 속성 변경
- 권한 변경
- 검색
- 정렬
- 보기 모드 변경
- 주소창 경로 입력
- 우클릭 컨텍스트 메뉴

이 조작들이 단순 UI 기능이 아니라, 작전 수행을 위한 전술 선택지가 되어야 한다.

### 4.2 전투보다 작전 수행

ProjectFR에서 중요한 것은 보안 캐릭터를 모두 제거하는 것이 아니다.

중요한 것은 다음이다.

- 목표 파일을 안전하게 회수했는가?
- 로그를 충분히 조작했는가?
- Trace를 감당 가능한 수준으로 유지했는가?
- 탈출 경로를 확보했는가?
- 보안 모듈을 우회했는가, 교란했는가, 아니면 제거했는가?

보안 모듈 처치는 하나의 선택지일 뿐이며, 항상 최선의 선택이어서는 안 된다.

### 4.3 파일 작업 진행률이 전술의 중심

ProjectFR의 핵심 전술은 “적의 HP를 깎는 것”이 아니라, **작업이 완료될 때까지 버티고, 숨기고, 속이는 것**이다.

예시:

- payroll.csv 복사 중... 35%
- records.log 위조 중... 60%
- archive.zip 압축 중... 20%
- 권한 변경 중... 80%

파일 작업은 즉시 완료되지 않을 수 있다.

- 작은 파일은 1턴 또는 즉시 완료
- 큰 DB 파일은 여러 턴 소요
- 암호화 파일은 분석 후 작업 가능
- 보안 파일은 작업 중 Trace 증가
- 백신이 감시 중인 파일은 작업이 중단될 수 있음

### 4.4 로그라이크 빌드 다양성

반복 플레이의 재미는 단순 능력치 증가가 아니라, 파일 조작 방식 자체가 바뀌는 데서 나온다.

좋은 업그레이드는 다음과 같다.

- 복사 시 미끼 파일 자동 생성
- 붙여넣기 시 감염 코드 확산
- 삭제한 파일이 휴지통 대신 바로 완전 삭제됨
- 압축 파일 내부는 스캔에 덜 노출됨
- 검색 사용 시 Trace 증가량 감소
- 클립보드에 든 파일은 감시 대상에서 제외됨
- 로그 위조가 주변 폴더까지 함께 적용됨
- 숨김 속성 파일은 Guard Scanner의 감시 우선순위에서 제외됨

ProjectFR의 빌드는 수치보다 행동 규칙을 바꿔야 한다.

### 4.5 파일 시스템 판타지 유지

모든 요소는 파일/폴더/운영체제 문법 안에서 설명되어야 한다.

- 폴더는 구역이다.
- 파일은 목표물, 자산, 함정, 흔적, 보안 객체다.
- 보안 모듈은 시스템 프로세스가 캐릭터처럼 시각화된 존재다.
- 플레이어는 커서형 침투 에이전트다.
- 액션은 스킬이 아니라 파일 조작 명령이다.
- 보스는 몬스터가 아니라 시스템 핵심 프로세스다.

## 5. Core Game Identity

기존의 일반적인 전투 게임 구조는 다음과 같다.  
적 발견 → 공격 → HP 감소 → 처치 → 보상

ProjectFR의 구조는 다음과 같아야 한다.  
목표 파일 발견 → 파일 작업 시작 → 보안 모듈 반응 → Trace 증가 → 교란/우회/스턴/로그 위조 → 작업 완료 → 탈출

핵심 정체성은 다음과 같다.

- 파일 = 목표물 / 자산 / 함정 / 흔적
- 폴더 = 침투 구역 / 보안 환경
- 보안 모듈 = 시스템 프로세스의 의인화
- 플레이어 = 커서형 침투 에이전트
- 전투 = 파일 작업을 성공시키기 위한 교란과 흔적 관리
- 승리 = 목표 달성 후 탈출

## 6. Primary Interaction Model

### 6.1 선택 후 명령

ProjectFR의 기본 조작은 **선택 후 명령**이다.

플레이어는 파일, 폴더, 보안 모듈, 빈 공간 중 하나를 선택하고, 가능한 명령을 실행한다.

예시:

- 파일 선택 → 복사 / 삭제 / 압축 / 속성 변경 / 로그 위조
- 폴더 선택 → 진입 / 스캔 / 숨김 파일 표시 / 권한 우회
- 보안 모듈 선택 → 관찰 / 기절 / 감염 / 유도 / 격리
- 빈 공간 선택 → 미끼 생성 / 붙여넣기 / 임시 경로 생성

이 방식은 실제 파일 탐색기의 사용 경험과 가장 잘 맞는다.

### 6.2 우클릭 컨텍스트 메뉴

ProjectFR에서 우클릭 메뉴는 단순 보조 UI가 아니라 핵심 전투 인터페이스다.

파일 우클릭 예시:

- 열기
- 복사
- 이동
- 삭제
- 이름 바꾸기
- 압축
- 속성
- 로그 위조
- 권한 분석
- 미끼로 복제

보안 모듈 우클릭 예시:

- 관찰
- 기절
- 우회
- 감염
- 미끼로 유도
- 격리
- 제거

폴더 우클릭 예시:

- 진입
- 스캔
- 숨김 파일 표시
- 방화벽 우회
- 임시 경로 생성
- 감시 해제

자주 쓰는 명령은 하단 퀵바에 배치한다.  
정밀 조작 = 우클릭 메뉴  
빠른 조작 = 하단 퀵바 / 단축키  
빌드 확장 = 메뉴 항목 추가 또는 변형

### 6.3 드래그 앤 드롭

파일 탐색기 기반 게임답게 드래그 앤 드롭도 핵심 조작이 될 수 있다.

예시:

- 목표 파일을 temp 폴더로 끌어 숨김
- 파일을 zip 안으로 드래그해 압축
- decoy 폴더로 미끼 파일 이동
- quarantine 밖으로 중요 파일 이동
- extraction zone으로 회수 파일 운반

드래그 앤 드롭은 직관적인 조작이지만, 게임적으로는 위험과 비용이 있어야 한다.

| 행동 | 효과 | 위험 |
|---|---|---|
| 파일을 다른 폴더로 드래그 | 빠른 이동 | 이동 로그 생성 |
| 파일을 zip에 드래그 | 은닉 | 압축 시간 발생 |
| 보안 파일을 드래그 | 강제 이동 | Trace 크게 증가 |
| 감시 중 파일을 드래그 | 작업 시도 | Guard 반응 |

## 7. Core System: File Operation Progress

### 7.1 작업 진행률

파일 작업은 많은 경우 즉시 끝나지 않고 진행률을 가진다.

- 복사 중... 40%
- 압축 중... 65%
- 로그 위조 중... 30%
- 권한 변경 중... 75%

진행 중인 작업은 보안 모듈의 방해를 받을 수 있다.

방해 예시:

- Guard Scanner가 작업 중인 파일 감시
- Indexer Scout가 최근 수정 파일 추적
- Antivirus Heavy가 위험 작업 강제 중단
- Backup Repairer가 삭제된 파일 복원 시도
- Firewall Sentinel이 폴더 이동 차단

### 7.2 작업 시간 결정 요소

작업 시간은 다음 요소로 결정된다.

- 파일 크기
- 파일 보안 등급
- 파일 속성
- 폴더 보안 레벨
- 현재 Trace
- 플레이어 빌드
- 사용한 툴
- 보안 모듈의 감시 여부

| 파일 | 작업 | 기본 소요 | 특이사항 |
|---|---|---|---|
| notes.txt | 복사 | 즉시 | Trace 낮음 |
| payroll.csv | 복사 | 2턴 | 목표 파일, 감시 대상 |
| employee.db | 복사 | 3턴 | 크기 큼 |
| secrets.enc | 접근 | 분석 필요 | 암호화 |
| records.log | 삭제 | 즉시 | Trace 크게 증가 |
| archive.zip | 압축/해제 | 2턴 | 은닉 가능 |

### 7.3 작업 중 보호와 교란

플레이어는 작업이 완료될 때까지 다양한 방법으로 시간을 벌 수 있다.

예시:

- Guard Scanner를 Stun
- Decoy 파일로 Indexer 유도
- logs 폴더의 records.log 위조
- target 파일을 hidden 속성으로 변경
- 백신 시야 밖으로 파일 이동
- temp 폴더를 경유해 Trace 감소
- 보안 모듈을 감염시켜 다른 보안 모듈 방해

이 구조가 ProjectFR의 전투를 만든다.

## 8. Time System Candidates

ProjectFR는 턴제, 실시간, 계획형 턴제 모두 가능하지만, 탐색기 기반 시스템과 가장 잘 맞는 방식은 다음이다.

### 8.1 추천 방식: 명령 큐 기반 실행

플레이어는 한 번에 여러 명령을 예약하고, 실행 버튼을 눌러 처리한다.

예시:

1. Guard Scanner 기절
2. payroll.csv 복사 시작
3. records.log 위조
4. temp 폴더로 이동

실행하면 짧은 애니메이션과 함께 순서대로 처리된다.

- 작업 1/4: Scanner.stun 실행 중...
- 작업 2/4: payroll.csv 복사 중...
- 작업 3/4: records.log 수정 중...
- 작업 4/4: temp 이동 중...

이 방식은 윈도우 작업 대기열과 해킹 스크립트 실행의 감각을 동시에 준다.

### 8.2 대안: 순수 턴제

장점:

- 전략적 판단이 명확함
- 로그라이크 빌드 조합을 이해하기 쉬움
- 구현 난이도가 상대적으로 안정적

단점:

- 정적으로 보일 수 있음
- 파일 탐색기 조작의 생동감이 약할 수 있음

### 8.3 대안: 실시간 + 일시정지

장점:

- 생동감이 강함
- 보안 모듈 순찰과 스캔 압박이 직관적
- 파일 작업 진행률과 잘 어울림

단점:

- 피지컬 중심으로 흐를 수 있음
- 빌드 조합보다 조작 속도가 중요해질 위험
- UI가 복잡해질 수 있음

### 8.4 최종 권장 방향

초기 프로토타입은 다음 방식을 권장한다.

> 명령 큐 기반 계획형 턴제

이 방식은 다음 장점을 가진다.

- 전략성 유지
- 캐릭터 연출 가능
- 파일 작업 진행률 표현 가능
- 로그라이크 빌드 조합과 잘 맞음
- 실시간처럼 보이지만 피지컬 게임으로 흐르지 않음

한 줄 원칙:

> 전략은 턴제처럼, 표현은 실시간처럼.

## 9. Windows Explorer Systems as Gameplay

### 9.1 보기 모드

탐색기의 보기 모드는 단순 UI 옵션이 아니라 전략 요소가 될 수 있다.

| 보기 모드 | 효과 | 단점 |
|---|---|---|
| 큰 아이콘 | 파일/보안 캐릭터 위치 파악 쉬움 | 세부 정보 부족 |
| 자세히 | 크기, 수정일, 권한, 위험도 확인 | 전술 배치 가독성 감소 |
| 목록 | 많은 파일을 빠르게 확인 | 위험 표현 약함 |
| 트리 | 전체 경로와 우회 루트 파악 | 현재 폴더 전투 정보 약함 |
| 숨김 파일 표시 | 숨겨진 목표 발견 | Trace 증가 또는 Indexer 반응 |

### 9.2 정렬과 필터링

정렬도 전술이 될 수 있다.

| 정렬 기준 | 게임 효과 |
|---|---|
| 이름순 | 목표 파일 탐색 쉬움 |
| 수정 날짜순 | 최근 변경된 로그/함정 추적 |
| 크기순 | 대용량 중요 데이터 식별 |
| 파일 형식순 | .log, .enc, .exe 분리 |
| 보안 등급순 | 위험 파일 우선 확인 |

필터링은 강력하지만 분석 행동으로 간주되어 Trace를 증가시킬 수 있다.

### 9.3 검색창

탐색기 검색창은 강력한 액션이다.  
검색: payroll

효과:

- 목표 파일 후보 표시
- 숨김 파일 일부 탐지
- 관련 로그 발견 가능

위험:

- Indexer Scout 활성화
- Trace 증가
- 검색 로그 생성
- 잘못된 키워드 입력 시 decoy 또는 honeypot에 낚일 수 있음

검색 특화 빌드가 가능하다.

### 9.4 주소창 경로 입력

주소창에 직접 경로를 입력하는 것도 게임 시스템이 될 수 있다.  
C:/Users/Admin/Finance/Payroll

성공 시:

- 목표 폴더로 빠르게 이동
- 우회 경로 확보

실패 시:

- 권한 오류
- honeypot 진입
- Trace 증가
- 보안 모듈 소환

### 9.5 속성 창

파일 속성은 중요한 전술 조작 대상이다.

속성 예시:

- 읽기 전용
- 숨김
- 시스템 파일
- 압축됨
- 암호화됨
- 최근 수정됨
- 소유자
- 권한
- 보안 등급

전술 예시:

- 목표 파일을 숨김 상태로 바꿔 감시 회피
- decoy를 시스템 파일로 위장
- 로그 파일을 읽기 전용으로 만들어 수정 차단
- 위험 파일을 압축 상태로 만들어 스캔 회피
- 파일 소유자를 바꿔 접근 허가 위조

### 9.6 휴지통

삭제는 끝이 아니라 휴지통으로 이동하는 행동이다.

| 행동 | 효과 | 위험 |
|---|---|---|
| Delete | 휴지통으로 이동 | 흔적 남음 |
| Shift + Delete | 완전 삭제 | Trace 크게 증가 |
| 휴지통 비우기 | 삭제 확정 | 보안 경고 발생 |
| 복원 | 삭제 취소 | Backup Repairer와 상호작용 |

### 9.7 클립보드

클립보드는 ProjectFR의 핵심 자원이 될 수 있다.

기본 규칙:

- 복사한 파일은 클립보드에 들어간다.
- 클립보드는 제한 용량을 가진다.
- 클립보드에 든 파일은 특정 빌드에서 스캔을 회피할 수 있다.
- 클립보드가 가득 차면 붙여넣기 전까지 추가 복사가 제한된다.

빌드 예시:

| 빌드 요소 | 효과 |
|---|---|
| Clipboard Buffer+ | 클립보드 용량 증가 |
| Ghost Clipboard | 클립보드 내 파일 Trace 감소 |
| Viral Paste | 붙여넣기 시 감염 확산 |
| Decoy Copy | 복사 시 원본 위치에 미끼 생성 |
| Memory Leak | 클립보드가 가득 차면 주변 보안 모듈 교란 |

### 9.8 멀티 윈도우 작전

ProjectFR의 멀티윈도우는 단순 연출이 아니라 **핵심 전술 시스템**이다.

기본 원칙:

- 해커는 한 창만 띄워두지 않는다.
- 각 창은 별도 작업 맥락과 역할을 가진다.
- 창을 늘릴수록 선택지는 늘지만 관리 리스크도 커진다.
- 멀티윈도우는 중후반 확장이 아니라, 초반부터 단계적으로 열리는 핵심 감각으로 본다.

창 예시:

- Main Infiltration Window
- Clipboard Window
- Temp Window
- Log Viewer Window
- Backup Window
- Archive Window

각 창의 역할:

| 창 | 역할 | 전술 의미 |
|---|---|---|
| Main Infiltration | 현재 침투 경로 탐색 | 기본 전술 공간 |
| Clipboard | 복사 대상 관리 | 회수/운반/은닉 흐름 분리 |
| Temp | 임시 저장/경유 | Trace 정리, 미끼 보관 |
| Log Viewer | 접근 기록 조작 | Search/Copy 후 흔적 관리 |
| Backup | 복원 서비스 대응 | 삭제/변조 후 원복 차단 |
| Archive | 압축 내부 조작 | 스캔 회피, 은닉 운반 |

운영 규칙:

- 동시에 열린 창 수는 자원이다.
- 포커스된 창에서만 직접 명령을 내릴 수 있다.
- 창 열기/전환/동시 유지에는 Trace 또는 Alert 비용이 붙을 수 있다.
- 일부 보안 모듈은 특정 창을 강제로 잠그거나 닫을 수 있다.

플레이 예시:  
main server 창에서 payroll.csv 복사 → clipboard 창에서 보관 확인 → temp 창으로 붙여넣기 → log viewer에서 접근 기록 위조 → backup 창에서 복원 프로세스 방해

1차 구현 우선순위:

1. Clipboard Window
2. Temp Window
3. Log Viewer Window

즉, 목표는 한 화면 안에서 명령만 누르는 것이 아니라,
**여러 창을 오가며 침투 흐름을 병렬 운영하는 해커식 작전 감각**을 만드는 것이다.

### 9.9 작업 관리자

보안 모듈은 화면에서는 캐릭터로 보이지만, 시스템적으로는 프로세스다.

작업 관리자 패널 예시:

- AntivirusCore.exe
- IndexerService.exe
- BackupRepairer.exe
- FirewallGate.sys
- AI_Monitor.dll

가능한 조작:

- 프로세스 종료
- 일시 중지
- 우선순위 낮추기
- 권한 탈취
- 가짜 프로세스로 위장
- 감염 코드 주입

## 10. Security Modules

보안 모듈은 몬스터가 아니라 시스템 프로세스의 시각화다.

| Firewall Sentinel | 방화벽 | 폴더 입구 차단, 권한 검사 |
| AI Monitor | 이상 행동 분석기 | 반복 패턴 감지, 고급 대응 |
| Permission Daemon | 권한 관리자 | 접근 권한 검증, 관리자 인증 요구 |

## 11. Security Response Flow

보안 시스템은 플레이어를 즉시 공격하기보다, 행동을 관찰하고 추적한다.  
행동 발생 → 로그 생성 → 이상 감지 → 추적 시작 → 위치 특정 → 격리 → 제거

보안 반응 단계:

| 단계 | 이름 | 효과 |
|---|---|---|
| 0 | Passive | 기본 순찰, 로그 수집 |
| 1 | Suspicious | 의심 폴더 감시, 스캔 주기 단축 |
| 2 | Active Scan | 최근 조작 파일 추적, 검색 로그 분석 |
| 3 | Quarantine | 폴더 봉쇄, 이동 경로 차단 |
| 4 | Purge | 플레이어 제거 시도, 감염 파일 삭제 |

## 12. Trace / Heat

### 12.1 Trace

Trace는 현재 작전 중 남긴 흔적이다.

Trace 증가 행동:

- 파일 삭제
- 시스템 파일 수정
- 대량 복사
- 검색 사용
- 주소창 직접 이동 실패
- 권한 변경
- 감염 확산
- 보안 파일 직접 접근
- 같은 폴더에서 반복 행동

Trace 감소 또는 우회 행동:

- 로그 위조
- temp 폴더 경유
- 압축 은닉
- 미끼 생성
- 숨김 속성 적용
- 보안 모듈 스턴
- 작업 관리자에서 프로세스 일시 중지

### 12.2 Heat

Heat는 캠페인 단위의 장기 추적도다.

Heat 증가 원인:

- 미션 실패
- 높은 Trace로 탈출
- 제한 턴 초과
- 완전 삭제 남발
- 보안 모듈 대량 제거
- 위험 의뢰 수락

Heat 효과:

| Heat 구간 | 효과 |
|---|---|
| 0-2 | 기본 상태 |
| 3-5 | 스캔 주기 단축, 보안 모듈 능력 소폭 증가 |
| 6+ | 고급 보안 모듈 등장, 제한 턴 감소, 보상 대비 위험 증가 |

Heat는 단순 페널티만이 아니라 빌드 자원이 될 수도 있다.

## 17. Build Archetypes

### 17.1 Ghost Build

흔적 최소화 빌드.

- Trace 증가 감소
- 숨김 속성 활용
- 로그 위조 강화
- temp 폴더 경유 보너스
- 보안 모듈과 직접 충돌 회피

### 17.2 Clipboard Build

복사/붙여넣기/클립보드 중심 빌드.

- 클립보드 용량 증가
- 복사 시 decoy 생성
- 붙여넣기 시 감염 또는 로그 위조
- 클립보드 내부 파일 스캔 회피

### 17.3 Virus Build

감염과 확산 중심 빌드.

- Inject 강화
- 감염된 보안 모듈이 주변 적 방해
- 붙여넣기 시 바이러스 확산
- 높은 Heat에서 강력해짐

### 17.4 Decoy Build

미끼와 유도 중심 빌드.

- 가짜 파일 생성
- 가짜 Trace 생성
- 보안 모듈 경로 변경
- 검색 결과 조작

### 17.5 Archive Build

압축/은닉/운반 중심 빌드.

- zip 내부 스캔 회피
- 압축 작업 시간 감소
- 압축 파일 안에 목표 숨기기
- 큰 파일 운반 효율 증가

### 17.6 Admin Build

권한 탈취와 정면 조작 중심 빌드.

- Permission Override 강화
- 잠긴 폴더 접근
- 보안 모듈 일시 정지
- 높은 Trace를 감수하고 빠르게 돌파

### 17.7 Search / Analyst Build

검색, 정렬, 정보 분석 중심 빌드.

- 검색 Trace 감소
- 위험 파일 자동 표시
- 숨김 파일 탐지
- 정렬/필터 사용 시 보너스
- 목표 경로 추론 강화

## 18. UI Layout Direction

### 18.1 기본 화면

ProjectFR의 전투 화면은 실제 윈도우 탐색기와 최대한 유사해야 한다.

핵심 구성:

- 상단: 리본 메뉴
- 상단 중간: 주소창
- 좌측: 폴더 트리 / 빠른 접근 / 드라이브 목록
- 중앙: 큰 아이콘 보기 기반의 파일/폴더 전술 공간
- 우측: 선택된 항목 정보 / 미션 목표 / 보안 반응
- 하단: 명령 퀵바 / 작업 큐 / 클립보드 상태

### 18.2 중앙 전술 공간

중앙 영역은 “필드”가 아니라 **큰 아이콘 보기 탐색기**처럼 보여야 한다.

원칙:

- 파일과 폴더는 격자 정렬
- 보안 캐릭터는 파일 앞/옆 슬롯에 배치
- 플레이어는 커서형 에이전트로 표시
- 이동 경로는 명령 예약 중에만 표시
- 시야 범위는 적 선택 또는 위험 예측 시에만 표시
- 목표 파일은 탐색기 선택 박스처럼 강조

중요한 점:

> 탐색기 배경 위에서 싸우는 것이 아니라, 탐색기 자체가 전술 보드가 되어야 한다.

### 18.3 우측 패널

우측 패널은 다음 정보를 보여준다.

- 현재 미션 목표
- 선택된 파일 정보
- 선택된 보안 모듈 정보
- 현재 Trace 원인
- 다음 보안 반응 예고
- 진행 중인 파일 작업

### 18.4 하단 패널

하단 패널은 다음을 포함한다.

- 주요 명령 퀵바
- 현재 AP 또는 명령 슬롯
- 작업 큐
- 클립보드 내용
- 실행 버튼
- 턴 종료 또는 명령 실행 버튼

예시:

작업 큐  
1.Scanner 기절  
2. payroll.csv 복사  
3. records.log 위조  
4. temp 폴더로 이동

[명령 실행]

## 19. Moment-to-Moment Example

미션 목표:  
/corp/server/finance/payroll.csv 회수

상황:

- payroll.csv는 Guard Scanner가 감시 중
- 복사에는 2턴 필요
- records.log가 접근 기록을 남김
- 3턴 뒤 전체 스캔 예정
- Backup Repairer가 삭제 파일을 복원할 수 있음

플레이어 선택:

1. Guard Scanner에게 Decoy 사용
2. payroll.csv 복사 시작
3. records.log Rewrite Log 예약
4. payroll.csv를 clipboard에 보관
5. temp 폴더로 이동 후 탈출

결과:

- Guard Scanner가 decoy.dat 쪽으로 이동
- payroll.csv 복사 100% 완료
- records.log 위조 성공
- Trace +4%
- 목표 파일 회수 완료
- 탈출 가능

이것이 ProjectFR의 이상적인 플레이 감각이다.

## 20. Narrative Direction

ProjectFR의 스토리는 미션 단위로 쌓인다.

초반:

- 기업 문서 회수
- 로그 삭제
- 내부 파일 정리
- 손상된 데이터 복구

중반:

- 내부 고발 자료 확보
- 경쟁사 sabotage
- 감염 파일 제거
- 숨겨진 백업 서버 침투

후반:

- 정치/군사/AI 통제 시스템 침투
- 진실 은폐 또는 폭로
- 의뢰인의 거짓말 확인
- 회수한 자료를 넘길지, 숨길지, 공개할지 선택

좋은 스토리 훅:

- 의뢰인이 항상 진실을 말하지 않음
- 같은 파일도 누군가에게는 증거, 누군가에게는 생명줄
- 삭제 의뢰가 사실은 진실 은폐일 수 있음
- 목표 파일이 실제 목표가 아닐 수 있음
- 백업 폴더에 숨겨진 진실이 남아 있을 수 있음

## 22. Final Direction

ProjectFR는 단순히 파일 탐색기처럼 보이는 게임이 아니다.

목표는 다음이다.

> 파일 탐색기를 플레이하는 로그라이크.

가장 중요한 원칙은 다음과 같다.

우클릭 메뉴  
+ 파일 작업 진행률  
+ 클립보드  
+ 권한/속성 조작  
+ 검색/정렬/보기 모드  
+ 보안 모듈 반응  
= ProjectFR의 핵심 전투 시스템

ProjectFR에서 플레이어는 몬스터를 죽이는 것이 아니라, 파일 시스템을 조작한다.

보안 모듈은 적이지만, 동시에 시스템 반응이다.

파일 작업은 공격보다 중요하다.

삭제는 만능 공격기가 아니라 위험한 선택이다.

복사, 압축, 로그 위조, 미끼, 권한 변경, 검색, 정렬, 속성 조작이 모두 전술이다.

최종적으로 ProjectFR는 다음 감각을 목표로 한다.

> “나는 던전을 도는 것이 아니라, 위험한 컴퓨터 안에서 실제로 파일을 조작하며 작전을 수행하고 있다.”
