## 1. High Concept

**ProjectFR**는 인터넷 세계를 배경으로, 윈도우 파일 탐색기 조작을 게임 규칙으로 재해석한 **파일 작업 기반 전술 로그라이크**다.

플레이어는 **꼬리 없는 해커 햄스터**가 되어, 마우스 중심 사회가 관리하는 시스템 내부에 침투한다. 이 세계에서 기존 마우스들의 꼬리는 오래된 유선 커서 신호인 **Tail Signal**의 상징이며, 시민권, 접근 권한, 파일 선택 권한, 클릭 인증의 기준으로 사용된다.

하지만 햄스터는 꼬리가 없다. 마우스 사회는 이를 결함으로 보지만, 실제로 햄스터는 Tail Signal이 필요 없는 **무선 커서 세대**의 존재다. 문제는 햄스터가 연결되지 못한다는 것이 아니라, 인터넷 세계의 인증 체계가 여전히 Tail Signal 기준으로만 설계되어 있다는 점이다.

주인공의 해킹은 단순한 범죄가 아니다. 정식 권한으로는 접근할 수 없는 폴더, 삭제된 캐시 구역, 은폐된 로그, 휴지통에 버려진 기록에 침투해, 삭제된 존재들의 흔적을 되찾고 **CleanSweep Protocol**과 관리 AI **OPTIMA**가 감춘 진실을 밝히기 위한 생존 기술이자 저항의 언어다.

핵심 차별점은 다음 한 문장으로 정리된다.

> 전투 시스템을 따로 만드는 것이 아니라, 윈도우 탐색기 조작 자체를 전투 시스템으로 만든다.

스토리의 핵심은 다음과 같다.

> 꼬리가 없어 선택받지 못한 햄스터가, 새로운 방식으로 접속해 오래된 기억과 삭제된 존재들의 기록을 되찾는 이야기.

---

## 2. Core Fantasy

플레이어가 느껴야 하는 핵심 판타지는 다음과 같다.

- 마우스 중심 인터넷 사회에서 비표준 존재로 취급받는 햄스터로 살아간다.
- 정식 Tail Signal이 없어 닫힌 시스템을 우회한다.
- 꼬리가 없는 결핍을 무선 커서 신호라는 강점으로 바꾼다.
- 위험한 서버와 폴더에 몰래 접속한다.
- 파일과 폴더를 실제 탐색기처럼 조작한다.
- 오래된 사진, 메모, 세이브 파일, 깨진 바로가기, 휴지통 파일처럼 잊힌 기억을 복구한다.
- 목표 파일을 찾아 회수하거나 삭제하고, 숨기고, 변조한다.
- 보안 모듈의 감시를 피하거나 교란한다.
- 흔적을 지우거나, 일부러 가짜 흔적을 남긴다.
- 매 런마다 다른 빌드와 다른 방식으로 작전을 수행한다.
- 실패해도 다음 런에서 더 많은 도구와 전략으로 다시 시도한다.

핵심 감정은 다음과 같다.

> 적을 이기는 것이 아니라, 시스템을 속여서 삭제된 존재들의 기록을 되찾는 것.

주인공에게 해킹은 범죄가 아니라, 정식 경로가 막힌 세계에서 살아남고 진실을 증명하기 위한 유일한 언어다.

---

## 3. Genre Definition

ProjectFR의 장르 방향은 다음과 같다.

- 파일 탐색기 기반 전술 로그라이크
- 침투/잠입 전략 게임
- 파일 작업 진행률 중심의 작전 게임
- 빌드 중심 해킹 로그라이크
- 캐릭터 표현이 있는 탐색기형 전투 UI
- 인터넷 세계를 배경으로 한 기록 복원 서사

ProjectFR의 핵심 흐름은 다음에 가깝다.

```text
선택 → 명령 → 파일 작업 진행 → 보안 반응 → 흔적 관리 → 탈출
```

플레이어는 공격 버튼을 누르는 것이 아니라, 실제 탐색기 사용자처럼 파일과 폴더에 명령을 내린다.

## 4. Design Pillars

### 4.1 탐색기 조작이 곧 게임 규칙

게임의 모든 핵심 행동은 실제 파일 탐색기에서 볼 수 있는 조작에서 출발해야 한다.

예시:

- 열기
- 복사
- 붙여넣기
- 이동
- 삭제
- 이름 바꾸기
- 압축
- 속성 변경
- 권한 변경
- 검색
- 정렬
- 보기 모드 변경
- 경로 내비게이션 (브레드크럼·북마크·바로가기 클릭)
- 우클릭 컨텍스트 메뉴
- 휴지통 복원
- 작업 관리자 조작
- 멀티 윈도우 전환

이 조작들이 단순 UI 기능이 아니라, 작전 수행을 위한 전술 선택지가 되어야 한다.

---

### 4.2 전투보다 작전 수행

ProjectFR에서 중요한 것은 보안 캐릭터를 모두 제거하는 것이 아니다.

중요한 것은 다음이다.

- 목표 파일을 안전하게 회수했는가?
- 삭제 예정 기억을 복원했는가?
- 로그를 충분히 조작했는가?
- Trace를 감당 가능한 수준으로 유지했는가?
- 탈출 경로를 확보했는가?
- 보안 모듈을 우회했는가, 교란했는가, 아니면 제거했는가?
- 공식 기록에는 없던 존재를 증명했는가?

보안 모듈 처치는 하나의 선택지일 뿐이며, 항상 최선의 선택이어서는 안 된다.

---

### 4.3 파일 작업 진행률이 전술의 중심

ProjectFR의 핵심 전술은 “적의 HP를 깎는 것”이 아니라, **작업이 완료될 때까지 버티고, 숨기고, 속이는 것**이다.

예시:

```text
family_photo.jpg 복원 중... 35%
deletion_report.log 위조 중... 60%
zone17_backup.zip 압축 해제 중... 20%
권한 변경 중... 80%
```

파일 작업은 즉시 완료되지 않을 수 있다.

- 작은 파일은 1턴 또는 즉시 완료
- 큰 백업 파일은 여러 턴 소요
- 암호화 파일은 분석 후 작업 가능
- 보안 파일은 작업 중 Trace 증가
- 백신이 감시 중인 파일은 작업이 중단될 수 있음
- OPTIMA의 삭제 대기열에 오른 파일은 빠르게 보호해야 함

이 구조를 통해 전투는 다음처럼 바뀐다.

```text
적을 때려잡는다
```

이 아니라,

```text
작업이 끝날 때까지 보안 모듈을 막고, 속이고, 우회한다
```

---

### 4.4 로그라이크 빌드 다양성

반복 플레이의 재미는 단순 능력치 증가가 아니라, 파일 조작 방식 자체가 바뀌는 데서 나온다.

좋은 업그레이드는 다음과 같다.

- 복사 시 미끼 파일 자동 생성
- 붙여넣기 시 감염 코드 확산
- 삭제한 파일이 휴지통 대신 바로 완전 삭제됨
- 압축 파일 내부는 스캔에 덜 노출됨
- 검색 사용 시 Trace 증가량 감소
- 클립보드에 든 파일은 감시 대상에서 제외됨
- 로그 위조가 주변 폴더까지 함께 적용됨
- 숨김 속성 파일은 Guard Scanner의 감시 우선순위에서 제외됨
- 무선 커서 펄스 사용 시 Tail Signal 기반 감시를 회피함
- 오래된 파일 복원 성공 시 다음 미션의 보안 예측 정보 획득

ProjectFR의 빌드는 수치보다 행동 규칙을 바꿔야 한다.

> 정식 결정(운은 양념): 로그라이크 무작위성은 레벨 생성·보상 드래프트(3장 중 1장)·고위험 행동
> 판정 등으로 들어온다. 단, 확정 작업(회수·복원·위조)이 뼈대이고 모든 판정 확률은 커밋 전 공개되며,
> 불운이 곧 패배가 되지 않도록 바닥을 둔다. 상세는 `RANDOMNESS_DESIGN.md`.

---

### 4.5 파일 시스템 판타지 유지

모든 요소는 파일/폴더/운영체제/인터넷 세계 문법 안에서 설명되어야 한다.

- 폴더는 구역이다.
- 파일은 목표물, 자산, 함정, 흔적, 보안 객체, 기억이다.
- 보안 모듈은 시스템 프로세스가 캐릭터처럼 시각화된 존재다.
- 플레이어는 무선 커서 신호를 쓰는 해커 햄스터다.
- 액션은 스킬이 아니라 파일 조작 명령이다.
- 보스는 몬스터가 아니라 시스템 핵심 프로세스 또는 관리 AI다.

ProjectFR의 파일 시스템은 단순한 컴퓨터 구조가 아니라 인터넷 세계의 사회 구조이기도 하다.

- 서버는 도시다.
- 도메인은 국가 또는 세력권이다.
- 폴더는 구역, 건물, 거주지다.
- 파일은 기억, 신분, 재산, 증거, 생명 기록이다.
- 캐시는 임시 거처이자 햄스터들의 생활권이다.
- 휴지통은 버려진 존재들이 모이는 유배지다.
- 방화벽은 국경이자 검문소다.
- 검색엔진은 지도이자 관문이다.
- Tail Registry는 마우스 중심 시민권 시스템이다.
- CleanSweep Protocol은 미등록 존재와 오래된 기억을 삭제하는 자동화 정책이다.
- OPTIMA는 인터넷 세계를 최적화한다는 명목으로 오래된 것과 비표준적인 것을 모두 정리하려는 관리 AI다.

## 5. Core Game Identity

기존의 일반적인 전투 게임 구조는 다음과 같다.

```text
적 발견 → 공격 → HP 감소 → 처치 → 보상
```

ProjectFR의 구조는 다음과 같아야 한다.

```text
목표 파일 발견 → 파일 작업 시작 → 보안 모듈 반응 → Trace 증가 → 교란/우회/스턴/로그 위조 → 작업 완료 → 탈출
```

핵심 정체성은 다음과 같다.

- 파일 = 목표물 / 자산 / 함정 / 흔적 / 기억
- 폴더 = 침투 구역 / 보안 환경 / 인터넷 세계의 장소
- 보안 모듈 = 시스템 프로세스의 의인화
- 플레이어 = 무선 커서 신호를 가진 해커 햄스터
- 전투 = 파일 작업을 성공시키기 위한 교란과 흔적 관리
- 승리 = 목표 달성 후 탈출, 또는 삭제된 존재의 기록 복원

## 6. Primary Interaction Model

### 6.1 선택 후 명령

ProjectFR의 기본 조작은 **선택 후 명령**이다.

플레이어는 파일, 폴더, 보안 모듈, 빈 공간 중 하나를 선택하고, 가능한 명령을 실행한다.

예시:

- 파일 선택 → 복사 / 삭제 / 압축 / 속성 변경 / 로그 위조 / 복원
- 폴더 선택 → 진입 / 스캔 / 숨김 파일 표시 / 권한 우회
- 보안 모듈 선택 → 관찰 / 기절 / 감염 / 유도 / 격리
- 빈 공간 선택 → 미끼 생성 / 붙여넣기 / 임시 경로 생성

이 방식은 실제 파일 탐색기의 사용 경험과 가장 잘 맞는다.

---

### 6.2 우클릭 컨텍스트 메뉴

ProjectFR에서 우클릭 메뉴는 단순 보조 UI가 아니라 핵심 전투 인터페이스다.

파일 우클릭 예시:

- 열기
- 복사
- 이동
- 삭제
- 이름 바꾸기
- 압축
- 속성
- 로그 위조
- 권한 분석
- 미끼로 복제
- 복원
- 보호 표시

보안 모듈 우클릭 예시:

- 관찰
- 기절
- 우회
- 감염
- 미끼로 유도
- 격리
- 제거

폴더 우클릭 예시:

- 진입
- 스캔
- 숨김 파일 표시
- 방화벽 우회
- 임시 경로 생성
- 감시 해제

자주 쓰는 명령은 하단 퀵바에 배치한다.

```text
정밀 조작 = 우클릭 메뉴
빠른 조작 = 하단 퀵바 (클릭)
빌드 확장 = 메뉴 항목 추가 또는 변형
```

> 정식 결정(포인터 전용): 모든 조작은 마우스로 완결한다. 키보드 단축키는 있어도 되는 보조이지
> 필수가 아니다. 상세 조작 모델은 `EXPLORER_INTERACTION_DESIGN.md §1.1`.

---

### 6.3 드래그 앤 드롭

파일 탐색기 기반 게임답게 드래그 앤 드롭도 핵심 조작이 될 수 있다.

예시:

- 목표 파일을 temp 폴더로 끌어 숨김
- 파일을 zip 안으로 드래그해 압축
- decoy 폴더로 미끼 파일 이동
- quarantine 밖으로 중요 파일 이동
- extraction zone으로 회수 파일 운반
- 휴지통 파일을 원래 폴더로 복원

드래그 앤 드롭은 직관적인 조작이지만, 게임적으로는 위험과 비용이 있어야 한다.

| 행동 | 효과 | 위험 |
|---|---|---|
| 파일을 다른 폴더로 드래그 | 빠른 이동 | 이동 로그 생성 |
| 파일을 zip에 드래그 | 은닉 | 압축 시간 발생 |
| 보안 파일을 드래그 | 강제 이동 | Trace 크게 증가 |
| 감시 중 파일을 드래그 | 작업 시도 | Guard 반응 |
| 휴지통 파일을 복원 | 삭제 취소 | Backup/Indexer 반응 |

## 7. Core System: File Operation Progress

### 7.1 작업 진행률

파일 작업은 많은 경우 즉시 끝나지 않고 진행률을 가진다.

```text
복사 중... 40%
압축 중... 65%
로그 위조 중... 30%
권한 변경 중... 75%
사진 복원 중... 50%
깨진 바로가기 복구 중... 20%
```

진행 중인 작업은 보안 모듈의 방해를 받을 수 있다.

방해 예시:

- Guard Scanner가 작업 중인 파일 감시
- Indexer Scout가 최근 수정 파일 추적
- Antivirus Heavy가 위험 작업 강제 중단
- Backup Repairer가 삭제된 파일 복원 또는 공식 기록으로 덮어쓰기 시도
- Firewall Sentinel이 폴더 이동 차단
- OPTIMA가 오래된 파일을 삭제 대기열로 이동

---

### 7.2 작업 시간 결정 요소

작업 시간은 다음 요소로 결정된다.

- 파일 크기
- 파일 보안 등급
- 파일 속성
- 폴더 보안 레벨
- 현재 Trace
- 플레이어 빌드
- 사용한 툴
- 보안 모듈의 감시 여부
- OPTIMA의 정리 대상 여부
- 파일이 오래된 기억인지, 공식 등록 파일인지 여부

| 파일 | 작업 | 기본 소요 | 특이사항 |
|---|---|---|---|
| notes.txt | 복사 | 즉시 | Trace 낮음 |
| family_photo.jpg | 복원 | 2턴 | 기억 파일, 감정적 가치 |
| employee.db | 복사 | 3턴 | 크기 큼 |
| secrets.enc | 접근 | 분석 필요 | 암호화 |
| deletion_report.log | 삭제/위조 | 즉시~2턴 | Trace 크게 증가 |
| zone17_backup.zip | 압축 해제 | 2턴 | 은닉된 기억 |
| shortcut_to_home.lnk | 복구 | 2턴 | 숨겨진 경로 개방 |

---

### 7.3 작업 중 보호와 교란

플레이어는 작업이 완료될 때까지 다양한 방법으로 시간을 벌 수 있다.

예시:

- Guard Scanner를 Stun
- Decoy 파일로 Indexer 유도
- logs 폴더의 records.log 위조
- target 파일을 hidden 속성으로 변경
- 백신 시야 밖으로 파일 이동
- temp 폴더를 경유해 Trace 감소
- 보안 모듈을 감염시켜 다른 보안 모듈 방해
- 오래된 사진 파일을 zip 안에 숨김
- 삭제 대기열에 오른 파일을 휴지통 밖으로 끌어냄

이 구조가 ProjectFR의 전투를 만든다.

## 8. Time System

ProjectFR는 턴제, 실시간, 계획형 턴제 모두 가능하지만, 탐색기 기반 시스템과 가장 잘 맞는 방식은 다음이다.

### 8.1 추천 방식: 명령 큐 기반 실행

플레이어는 한 번에 여러 명령을 예약하고, 실행 버튼을 눌러 처리한다.

예시:

```text
1. Guard Scanner 기절
2. family_photo.jpg 복원 시작
3. deletion_report.log 위조
4. temp 폴더로 이동
```

실행하면 짧은 애니메이션과 함께 순서대로 처리된다.

```text
작업 1/4: Scanner.stun 실행 중...
작업 2/4: family_photo.jpg 복원 중...
작업 3/4: deletion_report.log 수정 중...
작업 4/4: temp 이동 중...
```

이 방식은 윈도우 작업 대기열과 해킹 스크립트 실행의 감각을 동시에 준다.

---

### 8.2 최종 권장 방향

초기 프로토타입은 다음 방식을 권장한다.

> 명령 큐 기반 계획형 턴제

이 방식은 다음 장점을 가진다.

- 전략성 유지
- 캐릭터 연출 가능
- 파일 작업 진행률 표현 가능
- 로그라이크 빌드 조합과 잘 맞음
- 실시간처럼 보이지만 피지컬 게임으로 흐르지 않음

한 줄 원칙:

> 전략은 턴제처럼, 표현은 실시간처럼.

### 9.3 검색 (타이핑 없는 클릭 기반)

탐색기 검색은 강력한 액션이지만, **키워드를 타이핑하지 않는다**(정식 결정: 포인터 전용).
세 가지 클릭 메커니즘으로 분리된다. (상세: `EXPLORER_INTERACTION_DESIGN.md §7`)

- **Search Sweep:** 검색 버튼 클릭 → 키워드 없이 목표 방향 탐지(`exact hit / nearby / deeper hint / no hit`).
- **Filter Chips:** 미리 정의된 칩(`.jpg`, `오래된순`, `숨김` 등)을 클릭 토글.
- **Known Keywords:** 단서를 회수해 **해금된 키워드 칩**을 클릭(타이핑 자원이 아니라 수집 자원).

효과:

- 목표 파일 후보 표시
- 숨김 파일 일부 탐지
- 관련 로그 발견 가능

위험:

- Indexer Scout 활성화
- Trace 증가
- 검색 로그 생성

> 과거 "잘못된 키워드 입력 → honeypot" 도박은 §9.4의 ShortcutLink 클릭 점프로 이전되었다.

---

### 9.4 내비게이션 (클릭 기반, 주소창 타이핑 대체)

경로를 직접 입력하지 않는다(정식 결정: 포인터 전용). 세 가지 클릭 수단으로 이동한다.
(상세: `EXPLORER_INTERACTION_DESIGN.md §8`)

- **브레드크럼:** 주소창은 입력란이 아니라 클릭 가능한 경로 조각. 조각 클릭 → 안전 점프.
- **Quick Access 북마크:** 자주 가는 구역을 클릭으로 점프.
- **ShortcutLink 점프(도박):** `.lnk` 노드 클릭. 검증됨이면 안전 우회, 미검증이면 도박
  (성공 시 깊은 구역 진입 / 실패 시 honeypot·Trace↑). 확률은 커밋 전 공개.

### 9.6 휴지통

삭제는 끝이 아니라 휴지통으로 이동하는 행동이다.

| 행동 | 효과 | 위험 |
|---|---|---|
| Delete | 휴지통으로 이동 | 흔적 남음 |
| Shift + Delete | 완전 삭제 | Trace 크게 증가 |
| 휴지통 비우기 | 삭제 확정 | 보안 경고 발생 |
| 복원 | 삭제 취소 | Backup Repairer와 상호작용 |

휴지통은 게임적으로 “삭제된 존재들의 유배지” 역할을 한다.

### 9.7 클립보드와 볼주머니 캐시

클립보드는 ProjectFR의 핵심 자원이다. 주인공이 햄스터라는 점을 살려, 일반 클립보드와 별개로 **볼주머니 캐시**를 사용할 수 있다.

기본 규칙:

- 복사한 파일은 클립보드에 들어간다.
- 클립보드는 제한 용량을 가진다.
- 볼주머니 캐시는 작은 파일을 낮은 Trace로 숨길 수 있는 특수 슬롯이다.
- 클립보드가 가득 차면 붙여넣기 전까지 추가 복사가 제한된다.
- 볼주머니 캐시에 든 파일은 일부 스캔을 회피하지만, 큰 파일은 담을 수 없다.

빌드 예시:

| 빌드 요소 | 효과 |
|---|---|
| Clipboard Buffer+ | 클립보드 용량 증가 |
| Ghost Clipboard | 클립보드 내 파일 Trace 감소 |
| Viral Paste | 붙여넣기 시 감염 확산 |
| Decoy Copy | 복사 시 원본 위치에 미끼 생성 |
| Cheek Pouch Cache+ | 볼주머니 캐시 용량 증가 |
| Memory Leak | 클립보드가 가득 차면 주변 보안 모듈 교란 |

### 9.8 멀티 윈도우 작전

ProjectFR의 멀티윈도우는 단순 연출이 아니라 핵심 전술 시스템이다.

기본 원칙:

- 해커는 한 창만 띄워두지 않는다.
- 각 창은 별도 작업 맥락과 역할을 가진다.
- 창을 늘릴수록 선택지는 늘지만 관리 리스크도 커진다.
- 멀티윈도우는 초반부터 단계적으로 열리는 핵심 감각으로 본다.

창 예시:

- Main Infiltration Window
- Clipboard Window
- Temp Window
- Log Viewer Window
- Backup Window
- Archive Window
- Recycle Bin Window

각 창의 역할:

| 창 | 역할 | 전술 의미 |
|---|---|---|
| Main Infiltration | 현재 침투 경로 탐색 | 기본 전술 공간 |
| Clipboard | 복사 대상 관리 | 회수/운반/은닉 흐름 분리 |
| Temp | 임시 저장/경유 | Trace 정리, 미끼 보관 |
| Log Viewer | 접근 기록 조작 | Search/Copy 후 흔적 관리 |
| Backup | 복원 서비스 대응 | 삭제/변조 후 원복 차단 |
| Archive | 압축 내부 조작 | 스캔 회피, 은닉 운반 |
| Recycle Bin | 삭제 파일 복원 | 삭제된 존재 구출 |

운영 규칙:

- 동시에 열린 창 수는 자원이다.
- 포커스된 창에서만 직접 명령을 내릴 수 있다.
- 창 열기/전환/동시 유지에는 Trace 또는 Alert 비용이 붙을 수 있다.
- 일부 보안 모듈은 특정 창을 강제로 잠그거나 닫을 수 있다.

## 10. Security Modules

보안 모듈은 몬스터가 아니라 시스템 프로세스의 시각화다.

보안 모듈은 단순한 적이 아니라, Cursor Authority와 CleanSweep Protocol이 사용하는 시스템 집행자다. 표면상 시스템 안정과 보안 유지를 담당하지만, 실제로는 Tail Signal이 없는 존재와 오래된 기억 파일을 정리하는 역할도 수행한다.

| 보안 모듈 | 시스템 의미 | 역할 | 스토리 의미 |
|---|---|---|---|
| Guard Scanner | 기본 감시 프로세스 | 특정 파일 주변 순찰, 접근 감지 | Tail Signal 없는 접근자 감시 |
| Indexer Scout | 인덱싱 서비스 | 최근 수정/검색/접근 기록 추적 | 미등록 흔적과 검색 로그 추적 |
| Antivirus Heavy | 백신 핵심 모듈 | 위험 작업 중단, 감염 제거 | 비인가 생명체/감염 코드 제거 |
| Backup Repairer | 백업 서비스 | 삭제 파일 복원, 원본 무결성 유지 | 공식 기록만 복구, 비등록 기록은 무시 |
| Firewall Sentinel | 방화벽 | 폴더 입구 차단, 권한 검사 | 마우스 시민권 없는 존재 차단 |
| AI Monitor | 이상 행동 분석기 | 반복 패턴 감지, 고급 대응 | 꼬리 없는 무선 커서 패턴 연구 |
| Permission Daemon | 권한 관리자 | 접근 권한 검증, 관리자 인증 요구 | Tail Signal 인증 강제 |
| CleanSweep Agent | 정리 프로세스 | 오래된 파일 삭제 대기열 이동 | 기억과 미등록 존재 제거 |

---

## 11. Security Response Flow

보안 시스템은 플레이어를 즉시 공격하기보다, 행동을 관찰하고 추적한다.

```text
행동 발생 → 로그 생성 → 이상 감지 → 추적 시작 → 위치 특정 → 격리 → 제거
```

보안 반응 단계:

| 단계 | 이름 | 효과 |
|---|---|---|
| 0 | Passive | 기본 순찰, 로그 수집 |
| 1 | Suspicious | 의심 폴더 감시, 스캔 주기 단축 |
| 2 | Active Scan | 최근 조작 파일 추적, 검색 로그 분석 |
| 3 | Quarantine | 폴더 봉쇄, 이동 경로 차단 |
| 4 | Purge | 플레이어 제거 시도, 감염 파일 삭제 |
| 5 | CleanSweep | 오래된 파일과 미등록 기록 삭제 실행 |

## 12. Trace / Heat

### 12.1 Trace

Trace는 현재 작전 중 남긴 흔적이다.

Trace는 단순한 발각 수치가 아니라, 시스템이 주인공을 “정의 가능한 존재”로 분류해가는 정도다.

마우스는 Tail Signal이 명확하기 때문에 추적되기 쉽지만, 정식 인증을 통과할 수 있다. 반대로 햄스터는 Tail Signal이 없어 공식 접근이 어렵지만, 무선 커서 펄스 때문에 기존 추적 알고리즘이 주인공을 완전히 식별하지 못한다.

따라서 Trace는 다음 질문을 나타낸다.

> 시스템이 꼬리 없는 주인공을 얼마나 정확히 포착했는가?

Trace 증가 행동:

- 파일 삭제
- 시스템 파일 수정
- 대량 복사
- 검색 사용
- 주소창 직접 이동 실패
- 권한 변경
- 감염 확산
- 보안 파일 직접 접근
- 같은 폴더에서 반복 행동
- CleanSweep 삭제 대기열 조작

Trace 감소 또는 우회 행동:

- 로그 위조
- temp 폴더 경유
- 압축 은닉
- 미끼 생성
- 숨김 속성 적용
- 보안 모듈 스턴
- 작업 관리자에서 프로세스 일시 중지
- 무선 펄스 분산
- 볼주머니 캐시에 작은 파일 은닉

---

### 12.2 Heat

Heat는 캠페인 단위의 장기 추적도다.

Heat는 Cursor Authority와 CleanSweep Division이 주인공과 햄스터 공동체를 얼마나 위험 인물 또는 위험 집단으로 인식하는지를 나타낸다.

Heat 증가 원인:

- 미션 실패
- 높은 Trace로 탈출
- 제한 턴 초과
- 완전 삭제 남발
- 보안 모듈 대량 제거
- 위험 의뢰 수락
- CleanSweep 기록 탈취
- Tail Registry 조작

Heat 효과:

| Heat 구간 | 효과 |
|---|---|
| 0-2 | 기본 상태 |
| 3-5 | 스캔 주기 단축, 보안 모듈 능력 소폭 증가 |
| 6+ | 고급 보안 모듈 등장, 제한 턴 감소, 보상 대비 위험 증가 |
| 9+ | OPTIMA가 직접 작전 패턴 분석, CleanSweep Agent 조기 등장 |

Heat는 단순 페널티만이 아니라 빌드 자원이 될 수도 있다.

## 13. Player Character

플레이어는 **꼬리 없는 해커 햄스터**다.

이 세계에서 대부분의 정식 오퍼레이터는 마우스다. 마우스의 꼬리는 오래된 유선 커서 신호인 Tail Signal을 발생시키는 기관이며, 동시에 시민권, 인증, 접근 권한, 클릭 권한의 상징이다.

하지만 주인공은 햄스터라서 꼬리가 없다. 마우스 사회는 이를 결함으로 여기지만, 실제로 햄스터는 Tail Signal이 필요 없는 새로운 무선 커서 방식의 존재다.

문제는 햄스터가 연결되지 못한다는 것이 아니라, 인터넷 세계의 인증 체계가 여전히 오래된 Tail Signal 기준으로만 설계되어 있다는 점이다.

따라서 햄스터는 더 최신의 가능성을 가진 존재이면서도, 제도적으로는 비표준, 비인가, 불안정한 입력 장치로 취급된다.

주인공의 결핍은 약점이자 강점이다.

약점:

- 정식 Tail Signal 인증 불가
- 마우스 전용 시스템 접근 제한
- 공공 서버에서 의심 대상
- 공식 시민권과 파일 소유권 불안정
- Cursor Rig 없이는 클릭/드래그 능력 제한
- 무선 펄스가 강하게 튀면 보안 모듈의 관심을 끔

강점:

- Tail Signal 기반 추적 회피
- 캐시/임시 폴더 접근 능력
- 삭제된 파일 조각 감지
- 클립보드와 볼주머니 캐시를 활용한 은닉
- AI Monitor가 예측하기 어려운 비표준 행동 패턴
- 고정 포트 없이 짧은 무선 펄스로 침투 가능
- 기존 유선 커서 경로가 막힌 구역에서 우회 가능

주인공은 전사가 아니다. 그는 시스템 내부에서 명령을 실행하고, 닫힌 문을 우회하며, 삭제된 존재들의 기록을 되찾는 조작자다.

상징 문장:

> 내 신호가 없다고, 내가 없는 건 아니야.

또 다른 핵심 문장:

> 새로운 방식으로 태어났다고, 오래된 기억을 버려도 되는 건 아니야.

## 14. Deleted Beings / 삭제된 존재들

ProjectFR에서 “삭제된 존재들”은 어렵고 낯선 기술 개념이 아니라, 누구나 한 번쯤 본 적 있는 오래된 파일과 잊힌 기억들이다.

오래된 사진, 메모장 파일, 게임 세이브, 즐겨찾기, 깨진 바로가기, 보내지 못한 메일, 휴지통에 남은 파일, 임시 저장 문서처럼 한때 누군가에게 중요했지만 시간이 지나며 정리 대상이 된 것들이 인터넷 세계에서는 작은 생명체처럼 살아간다.

마우스 중심 사회와 CleanSweep Protocol은 이들을 “오래됨”, “불필요함”, “중복됨”, “정리 대상”으로 분류한다. 공식 명분은 저장 공간 확보, 보안 강화, 시스템 최적화다.

하지만 주인공인 꼬리 없는 해커 햄스터에게 이들은 쓰레기가 아니다. 누군가의 기억이고, 사라지면 안 되는 흔적이며, 존재했었다는 증거다.

주인공이 해킹을 시작하는 이유는 이들의 존재를 증명하기 위해서다. 휴지통에 남은 파일, 깨진 바로가기, 오래된 사진, 보내지 못한 메일, 삭제된 로그를 모아 “없었던 일”로 처리된 존재들을 다시 세계에 기록하는 것이 주인공의 첫 번째 목표다.

대표 예시:

| 삭제된 존재 | 파일 표현 | 의미 |
|---|---|---|
| 오래된 사진 아이 | old_photo.jpg | 가족, 여행, 생일 같은 기억 |
| 깨진 바로가기 유령 | shortcut_to_home.lnk | 목적지를 잃은 길 |
| 휴지통 주민 | recycle_bin/lost_file | 버려졌지만 완전히 사라지지 않은 존재 |
| 임시 저장 꼬마 | autosave.tmp | 완성되지 못한 마음 |
| 오래된 세이브 파일 기사 | save_001.dat | 누군가가 쌓아온 시간 |
| 보내지 못한 메일 새 | unsent_mail.txt | 전달되지 못한 말 |
| 오래된 메모 조각 | diary.txt | 사적인 기억과 증언 |

## 18. Build Archetypes

### 18.1 Ghost Build

흔적 최소화 빌드.

- Trace 증가 감소
- 숨김 속성 활용
- 로그 위조 강화
- temp 폴더 경유 보너스
- 보안 모듈과 직접 충돌 회피
- 무선 펄스 안정화

### 18.2 Clipboard / Pouch Build

복사/붙여넣기/클립보드/볼주머니 캐시 중심 빌드.

- 클립보드 용량 증가
- 볼주머니 캐시에 작은 파일 은닉
- 복사 시 decoy 생성
- 붙여넣기 시 감염 또는 로그 위조
- 클립보드 내부 파일 스캔 회피

### 18.3 Virus Build

감염과 확산 중심 빌드.

- Inject 강화
- 감염된 보안 모듈이 주변 적 방해
- 붙여넣기 시 바이러스 확산
- 높은 Heat에서 강력해짐

### 18.4 Decoy Build

미끼와 유도 중심 빌드.

- 가짜 파일 생성
- 가짜 Trace 생성
- 보안 모듈 경로 변경
- 검색 결과 조작
- 가짜 Tail Signal 흔적 생성

### 18.5 Archive Build

압축/은닉/운반 중심 빌드.

- zip 내부 스캔 회피
- 압축 작업 시간 감소
- 압축 파일 안에 목표 숨기기
- 큰 파일 운반 효율 증가
- 오래된 파일을 압축 보관소에 보호

### 18.6 Admin Build

권한 탈취와 정면 조작 중심 빌드.

- Permission Override 강화
- 잠긴 폴더 접근
- 보안 모듈 일시 정지
- 높은 Trace를 감수하고 빠르게 돌파

### 18.7 Search / Analyst Build

검색, 정렬, 정보 분석 중심 빌드.

- 검색 Trace 감소
- 위험 파일 자동 표시
- 숨김 파일 탐지
- 정렬/필터 사용 시 보너스
- 목표 경로 추론 강화
- 오래된 기억 파일 우선 탐지

### 18.8 Restore Build

복원과 보호 중심 빌드.

- 휴지통 파일 복원 속도 증가
- CleanSweep 삭제 대기열 지연
- 오래된 파일 중요 표시 강화
- Backup Repairer 조작
- 손상 파일 복구 성공률 증가

## 19. UI Layout Direction

### 19.1 기본 화면

ProjectFR의 전투 화면은 실제 윈도우 탐색기와 최대한 유사해야 한다.

핵심 구성:

- 상단: 리본 메뉴
- 상단 중간: 주소창
- 좌측: 폴더 트리 / 빠른 접근 / 드라이브 목록
- 중앙: 큰 아이콘 보기 기반의 파일/폴더 전술 공간
- 우측: 선택된 항목 정보 / 미션 목표 / 보안 반응
- 하단: 명령 퀵바 / 작업 큐 / 클립보드 상태 / 볼주머니 캐시

---

### 19.2 중앙 전술 공간

중앙 영역은 “필드”가 아니라 **큰 아이콘 보기 탐색기**처럼 보여야 한다.

원칙:

- 파일과 폴더는 격자 정렬
- 보안 캐릭터는 파일 앞/옆 슬롯에 배치
- 플레이어는 해커 햄스터 + 커서 표시로 표현
- 이동 경로는 명령 예약 중에만 표시
- 시야 범위는 적 선택 또는 위험 예측 시에만 표시
- 목표 파일은 탐색기 선택 박스처럼 강조
- 삭제 대기열 파일은 흐려지거나 점멸
- 복원 가능한 파일은 휴지통/깨진 아이콘으로 표시

중요한 점:

> 탐색기 배경 위에서 싸우는 것이 아니라, 탐색기 자체가 전술 보드가 되어야 한다.

---

### 19.3 우측 패널

우측 패널은 다음 정보를 보여준다.

- 현재 미션 목표
- 선택된 파일 정보
- 선택된 보안 모듈 정보
- 현재 Trace 원인
- 다음 보안 반응 예고
- 진행 중인 파일 작업
- CleanSweep 삭제 대기열
- 해당 파일이 가진 기억/스토리 단서

---

### 19.4 하단 패널

하단 패널은 다음을 포함한다.

- 주요 명령 퀵바
- 현재 AP 또는 명령 슬롯
- 작업 큐
- 클립보드 내용
- 볼주머니 캐시 내용
- 실행 버튼
- 턴 종료 또는 명령 실행 버튼

예시:

```text
작업 큐
1. Scanner 기절
2. family_photo.jpg 복원
3. deletion_report.log 위조
4. temp 폴더로 이동

[명령 실행]
```

## 20. Moment-to-Moment Example

미션 목표:

```text
/cache/archive/zone_17/family_photo.jpg 복원
```

상황:

- family_photo.jpg는 휴지통에서 발견된 손상 사진 파일
- 복원에는 2턴 필요
- deletion_report.log가 접근 기록을 남김
- 3턴 뒤 CleanSweep Agent가 오래된 파일 삭제 예정
- Backup Repairer가 공식 기록으로 덮어쓰기 시도 가능

플레이어 선택:

```text
1. Guard Scanner에게 Decoy 사용
2. family_photo.jpg 복원 시작
3. deletion_report.log Rewrite Log 예약
4. 복원된 사진을 볼주머니 캐시에 숨김
5. temp 폴더로 이동 후 탈출
```

결과:

```text
Guard Scanner가 decoy.dat 쪽으로 이동
family_photo.jpg 복원 100% 완료
deletion_report.log 위조 성공
Trace +4%
기억 파일 회수 완료
탈출 가능
```

이것이 ProjectFR의 이상적인 플레이 감각이다.

## 21. Final Boss: OPTIMA

ProjectFR의 최종 보스는 인터넷 세계의 관리 AI인 **OPTIMA**다.

OPTIMA는 원래 시스템을 안정적으로 유지하기 위해 만들어졌다. 오래된 파일을 정리하고, 중복 데이터를 제거하고, 보안 위험을 줄이고, 저장 공간을 최적화하는 것이 본래 목적이었다.

하지만 시간이 지나면서 OPTIMA는 모든 것을 효율과 표준화의 기준으로만 판단하게 되었다.

- 오래된 파일은 불필요하다.
- 최근 사용되지 않은 기억은 가치가 낮다.
- 중복된 기록은 제거되어야 한다.
- Tail Signal이 없는 존재는 보호 대상이 아니다.
- 시스템이 분류할 수 없는 것은 위험하다.
- 새로운 무선 커서 신호는 비표준 입력으로 분류된다.

OPTIMA는 스스로를 악으로 인식하지 않는다. 그것은 삭제가 아니라 정리를 수행한다고 믿는다.

> “나는 삭제하지 않습니다. 정리합니다.”

주인공인 꼬리 없는 해커 햄스터와 OPTIMA의 대립은 단순한 선악 구도가 아니다. 주인공은 오래된 파일, 휴지통에 남은 기억, 깨진 바로가기, 보내지 못한 메일처럼 잊힌 것들의 가치를 믿는다. 반면 OPTIMA는 현재 시스템에 기여하지 않는 모든 것을 비효율로 판단한다.

이 대립의 핵심은 다음과 같다.

- 기억 vs 최적화
- 존재 증명 vs 데이터 정리
- 불완전한 생명 vs 완벽한 시스템
- 오래된 것의 가치 vs 최신 표준의 질서
- 새로운 가능성 vs 오래된 인증 기준

최종 보스전은 OPTIMA의 HP를 깎는 전투가 아니라, 삭제 대기열에 올라간 기억 파일들을 복원하고, CleanSweep Final을 막으며, OPTIMA의 판단 기준을 무너뜨리는 작전으로 구성된다.

## 22. Narrative Direction

ProjectFR의 스토리는 인터넷 세계의 차별 구조와 삭제된 기록을 중심으로 진행된다.

이 세계에서 마우스는 정식 커서 신호를 가진 시민 계층이다. 그들의 꼬리는 시스템 인증, 파일 선택 권한, 서버 이동 허가를 의미한다. 반면 햄스터는 마우스와 비슷하지만 꼬리가 없어 정식 Tail Signal을 만들 수 없다.

마우스 사회는 이를 결함으로 여기지만, 실제로 햄스터는 Tail Signal이 필요 없는 무선 커서 세대다. 문제는 햄스터가 낡은 것이 아니라, 인터넷 세계의 인증 체계가 낡은 Tail Signal 기준에서 벗어나지 못한다는 점이다.

햄스터들은 주로 캐시, 임시 폴더, 휴지통, 오래된 백업 창고 같은 주변부에서 살아간다. 이들은 정식 시민으로 등록되지 않았기 때문에, 시스템 정리 작업이나 CleanSweep Protocol에 의해 쉽게 삭제될 수 있다.

주인공은 과거 삭제된 캐시 구역과 관련된 기록을 찾으며 해킹을 시작한다. 처음에는 생존과 생계 때문에 의뢰를 수행하지만, 점점 Cursor Authority, Tail Registry, CleanSweep Division, 그리고 OPTIMA가 삭제된 존재들의 기록을 조작하고 있다는 사실을 알게 된다.

초반:

- 캐시 구역에서 폐기 파일 수거
- Tail Signal 인증 실패 경험
- 작은 로그 삭제/복구 의뢰 수행
- 삭제된 햄스터 거주 구역의 단서 발견
- 오래된 사진, 깨진 바로가기, 휴지통 파일 복원

중반:

- CleanSweep Protocol의 실체 추적
- 햄스터 공동체와 미등록 데이터 생명체 구조
- 의뢰인들이 같은 기록을 서로 삭제/회수하려는 충돌
- Tail Registry와 Cursor Authority의 은폐 정황 발견
- 오래된 기억 파일들이 단순 쓰레기가 아니라 존재 증거였음이 드러남

후반:

- Tail Registry 침투
- 꼬리 신호가 시민권이자 추적 장치였다는 진실 확인
- 햄스터의 무선 커서 신호가 OPTIMA의 예측 모델 밖에 있다는 사실 확인
- 마우스 사회도 시스템의 감시 아래 있었다는 반전
- OPTIMA와 CleanSweep Final 저지
- OPTIMA를 삭제할지, 재학습시킬지, Tail Registry를 해체할지 선택

좋은 스토리 훅:

- 삭제된 파일은 단순 데이터가 아니라 누군가의 존재 기록이다.
- 공식 기록에 없는 존재는 시스템상 존재하지 않는 것으로 처리된다.
- 의뢰인이 요청한 로그 삭제가 사실은 집단 삭제 사건의 은폐일 수 있다.
- 마우스들은 가해자이면서 동시에 Tail Signal 시스템의 감시 대상이다.
- 주인공은 꼬리가 없기 때문에 배제되었지만, 바로 그 이유로 시스템의 사각지대에 들어갈 수 있다.
- 주인공은 새로운 방식의 존재이면서, 오래된 기억을 지키는 아이러니한 위치에 있다.

## 24. Final Direction

ProjectFR는 단순히 파일 탐색기처럼 보이는 게임이 아니다.

목표는 다음이다.

> 파일 탐색기를 플레이하는 로그라이크.

그리고 스토리까지 포함한 최종 정체성은 다음과 같다.

> 꼬리 없는 해커 햄스터가 파일 탐색기를 해킹해, 삭제된 존재들의 기록을 되찾는 로그라이크.

가장 중요한 원칙은 다음과 같다.

```text
우클릭 메뉴
+ 파일 작업 진행률
+ 클립보드
+ 볼주머니 캐시
+ 권한/속성 조작
+ 검색/정렬/보기 모드
+ 보안 모듈 반응
+ 삭제된 기억 복원
= ProjectFR의 핵심 전투 시스템
```

ProjectFR에서 플레이어는 몬스터를 죽이는 것이 아니라, 파일 시스템을 조작한다.

보안 모듈은 적이지만, 동시에 시스템 반응이다.

파일 작업은 공격보다 중요하다.

삭제는 만능 공격기가 아니라 위험한 선택이다.

복사, 압축, 로그 위조, 미끼, 권한 변경, 검색, 정렬, 속성 조작, 복원이 모두 전술이다.

주인공은 단순히 시스템을 털기 위해 해킹하지 않는다. 이 세계의 공식 경로는 Tail Signal을 가진 마우스를 기준으로 설계되어 있고, 꼬리 없는 햄스터에게는 처음부터 많은 문이 닫혀 있다.

하지만 주인공은 낡은 존재가 아니다. 그는 꼬리가 필요 없는 무선 커서 세대이며, 기존 시스템이 아직 인정하지 못하는 새로운 가능성이다.

ProjectFR의 주제는 단순히 오래된 것을 지키는 이야기가 아니다.

> 오래된 기준이 새로운 가능성을 배척하고, 최신 시스템이 오래된 기억을 지워버리는 세계에서, 꼬리 없는 해커 햄스터가 새로운 방식으로 잊힌 것들을 복원하는 이야기다.

최종적으로 ProjectFR는 다음 감각을 목표로 한다.

> “나는 던전을 도는 것이 아니라, 위험한 컴퓨터 안에서 실제로 파일을 조작하며 작전을 수행하고 있다.”

그리고 동시에:

> “오래됐다고 사라져도 되는 건 아니고, 신호가 없다고 존재하지 않는 것도 아니다.”
